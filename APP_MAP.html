Open questions / things to tweak
?
Backend vs. frontend split: infrared_runner.py is spawned as a child process by infrared.js — but infrared.js currently reads the pre-computed result file, not a live process. Is the intent to always run live, or should the pre-computed JSON stay as the default fallback?
?
Adding more festivals: Glastonbury + Untold are listed as "coming soon" cards. What changes: a new config.json entry, new pre-fetched GeoJSONs, and a new pre-computed UTCI grid per festival? Or should the festival selector trigger a live run on click?
?
State management (S{} object): all runtime state lives in a single global S{} in index.html. If adding multiple festivals or user-drawn zones, should this become a proper store (or at least per-festival slots)?
?
Planned factors #9 & #10: Pedestrian flow (OSM highway graph) and shade cost optimisation are marked as planned. Where do these live — new JS module, extension to placement.js, or a separate Python step?
?
tokens.js exposure: Mapbox token lives in tokens.js (browser-readable, in .gitignore). Current approach works for local dev, but what's the plan when this is deployed publicly?
